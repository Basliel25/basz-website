## The Problem

I'd been reading OSTEP's concurrency chapters and wanted something concrete to chew on before the MapReduce project. Reading about mutexes is fine. Watching a producer-consumer queue actually deadlock on your own machine is better.

So I spent a weekend writing **HiveParser** — a multithreaded log parser in C that eats a syslog-format file, classifies severity, and spits out an aggregated report. Eight worker threads, one shared queue, per-thread report structs that get merged at the end. Nothing fancy. The point wasn't to ship a tool. The point was to make every concurrency bug OSTEP warns about happen to me, on my machine, with my code, so I'd never forget what they look like.

It works. It also taught me more in three days than two weeks of reading did.

## Why a Log Parser

Log parsing is the textbook embarrassingly-parallel workload, and that's exactly why I picked it. Every line is independent. No shared state between lines. The only synchronisation you need is at the boundaries — handing lines to workers, and merging results at the end. If I couldn't get this right, I had no business attempting MapReduce next.

I grabbed the `Linux_2k.log` file from the [loghub](https://github.com/logpai/loghub) dataset — 2000 lines of real syslog output, the kind of thing `/var/log/syslog` looks like on any normal machine. Format is fixed-ish:

```
Jun 14 15:16:01 combo sshd(pam_unix)[19939]: authentication failure; ...
```

Month, day, timestamp, host, component, PID, message. The parser extracts those fields, pulls a severity tag out of the message body, and counts things.

## The Design

The architecture is a classic single-producer, multi-consumer queue. Main thread reads the file line by line and pushes onto a linked-list queue. Eight worker threads pop lines off, parse them, and update their *own* private report struct. When the file's done, main thread sets a `done` flag and broadcasts on the condvar. Workers drain the queue, exit, get joined. Main thread merges the eight per-thread reports into one global report and prints it.

```
file → [main thread enqueues] → [shared queue] → [8 workers dequeue + parse] → 8 per-thread reports
                                                                                       │
                                                                                       ▼
                                                                              [merge] → global report
```

The per-thread-then-merge pattern is the same one I used in the OSTEP sloppy counter exercise, and it's the same pattern that shows up in MapReduce's reduce phase. That repetition isn't an accident — get it into your fingers on a small project, and the bigger project stops feeling so big.

## The Queue

The queue itself is a textbook intrusive linked list with a head pointer, a tail pointer, a size counter, a mutex, a condvar, and a done flag:

```c
typedef struct __workQueue_t {
    struct Node *head;
    struct Node *tail;
    int size;
    pthread_mutex_t lock;
    pthread_cond_t cond;
    int flag;          /* 1 when producer is done, 0 otherwise */
} workQueue_t;
```

The first decision I had to make was who locks the queue. I went back and forth on this. The "obvious" thing is to have `enqueue()` and `dequeue()` lock the mutex themselves — encapsulation, looks clean. But the producer needs to do *enqueue + signal* atomically (otherwise you race the condvar), and the consumer needs to do *check size + wait + dequeue* atomically (otherwise you race past your own wait). If `enqueue` and `dequeue` lock internally, the caller has to lock too, and now you've got recursive lock pain or you've split the critical section in half and broken correctness.

So I made the rule explicit: **the caller holds the lock**. The queue functions assume it.

```c
void enqueue(workQueue_t *queue, char *line) {
    Node *new_node = malloc(sizeof(Node));
    new_node->line = line;
    new_node->next = NULL;

    /* Caller holds the lock — just manipulate the queue */
    if (!queue->tail) {
        queue->tail = queue->head = new_node;
    } else {
        queue->tail->next = new_node;
        queue->tail = new_node;
    }
    queue->size++;
}
```

That's not the prettiest API, but it's honest about who owns the synchronisation. The producer side then looks like this:

```c
while (fgets(buffer, sizeof(buffer), log) != NULL) {
    char *line = strdup(buffer);
    pthread_mutex_lock(&work_Queue->lock);
    enqueue(work_Queue, line);
    pthread_cond_signal(&work_Queue->cond);
    pthread_mutex_unlock(&work_Queue->lock);
}
```

Lock, enqueue, signal, unlock. One critical section per line. The signal wakes one worker if any are sleeping; if all eight are busy, it's a no-op and the worker will see the queue is non-empty next time it loops.

## The Worker

The worker loop is where most of the OSTEP lessons hit at once:

```c
void *worker_thread(void *arg) {
    thread_args_t *args = (thread_args_t *)arg;
    workQueue_t *work_queue = args->queue;
    char *line;

    while (1) {
        pthread_mutex_lock(&work_queue->lock);

        /* Wait until work arrives OR done flag is set */
        while (work_queue->size == 0 && !work_queue->flag) {
            pthread_cond_wait(&work_queue->cond, &work_queue->lock);
        }

        /* If queue is empty and producer is done, exit */
        if (work_queue->size == 0 && work_queue->flag) {
            pthread_mutex_unlock(&work_queue->lock);
            break;
        }

        line = dequeue(work_queue);
        pthread_mutex_unlock(&work_queue->lock);

        log_entry_t *entry = extract_entry(line);
        update_report(entry, &args->thread_report);
        free(entry);
        free(line);
    }
    return NULL;
}
```

A few things in there are worth dwelling on, because I got every one of them wrong at least once before they ended up looking like that.

**The `while` around `pthread_cond_wait`, not `if`.** This is the spurious-wakeup trap. Condvars can wake threads even when no signal was sent. If you check the predicate with `if`, a spurious wakeup walks straight past the check into a `dequeue()` on an empty queue and you crash. The `while` loop re-checks the predicate every time you wake, and only exits when it's actually true. OSTEP hammers this and now I know why.

**The done-flag pattern.** When the producer is finished, it sets `flag = 1` and `pthread_cond_broadcast`'s. Without `broadcast`, only one of the eight workers wakes, the other seven sleep on the condvar forever, and the program hangs. The first time I ran this with `pthread_cond_signal` instead of `broadcast`, the program just sat there and I had to `kill -9` it. Lovely lesson, will not repeat.

**Unlock before parsing.** The critical section is `dequeue()`, period. Once you've got the line in your local pointer, you drop the lock and parse outside. If you parse while holding the lock, your eight workers serialise on the parse and you've written a single-threaded program with extra steps. This is the whole point of the per-thread report — every worker is doing real work in parallel because the only contended resource is the queue itself, and contact with the queue is microseconds long.

## Per-Thread Reports

Every worker has its own `report_t` — severity counts, an hourly distribution array, earliest/latest timestamps. When a worker parses a line, it updates *its own* report, no locks involved:

```c
typedef struct __report_t {
    int total_log;
    int errors;
    int warn;
    int info;
    int hourly[24];
    double error_percentage;
    time_t earliest, latest;
} report_t;
```

After `pthread_join` returns for all eight workers — meaning every report is finalised and nobody's writing to it anymore — main thread merges them sequentially:

```c
void merge_report(report_t *master, report_t *thread) {
    master->total_log += thread->total_log;
    master->errors    += thread->errors;
    master->warn      += thread->warn;
    master->info      += thread->info;

    for (int i = 0; i < 24; i++) {
        master->hourly[i] += thread->hourly[i];
    }
    master->earliest = (thread->earliest < master->earliest)
                     ? thread->earliest : master->earliest;
    master->latest   = (thread->latest > master->latest)
                     ? thread->latest : master->latest;
}
```

This is the part I want to emphasise, because it's the bit that transfers directly to the next project. **No mutex.** No atomic operations. No contention. The merge is a single-threaded loop over eight structs that nobody else can possibly be touching, because their owning threads are dead. If you tried to do this with one shared global report and a mutex around every `errors++`, eight threads would spend most of their time waiting for the lock instead of parsing. The sloppy-counter chapter in OSTEP shows you the throughput curve. I've now seen it on my own machine.

> **Note:** `time_t` comparisons assume both timestamps are valid. The earliest field is initialised to `LLONG_MAX` in `report_init()` so the first real timestamp always wins. Sentinel values matter — without that init, the merge takes whatever garbage was on the stack as "earliest" and you get timestamps from 1973.

## The Bugs I Hit

Three of these are still in the repo as I write this. I'm fixing them as part of the v0.1 cleanup, but they're worth naming because they're the kind of things you'd hit too.

**Severity classification was case-broken.** The `severity()` function lowercases the message before doing `strstr`, then returns capitalised strings like `"Error"`, `"Warn"`, `"Info"`. Then `update_report` does `strstr(entry->severity, "Error")` — and that branch is checking for a capitalised substring inside a string that's *also* capitalised. Fine. But I had a bug where one branch was checking for `"failure"` twice (copy-paste in the chain of `if`s, second branch dead). Caught it on a code review pass. Stupid, easy to miss, exactly the kind of thing tests would have caught if I'd written them first.

**Pointers into freed memory.** `extract_entry` uses `strtok_r` on the input line, which returns pointers *into* that line. Then the worker frees the line right after the parse. Works only because the entry struct is also freed at the same time and nothing else touches it. It's the kind of fragile that bites you the moment someone refactors. v0.1 will `strdup` the fields out of the line so the entry owns its own memory.

**`static const int` for an array size at file scope.** I had `static const int NUMTHREADS = 8;` and used it as `pthread_t thread[NUMTHREADS]`. That works in modern C as a VLA, but VLAs at file scope aren't legal and they're a wart anyway. `#define NUMTHREADS 8` is the right move. Filed under "things King's book actually told me, that I forgot."

## What's Next

This is v0 — single-producer, fixed thread count, file-only input, stdout-only output, single global report at the end. Tier 2 is on the roadmap and looks like:

- **Top-N components by error rate.** A min-heap per thread, merged at the end. Same shape as the report merge, more interesting comparator.
- **Concurrent hashmap merge.** Right now everything fits in fixed counters. Tier 2 needs a real hashmap that multiple threads can write into safely, which is the actual hard concurrency problem.
- **Stream input.** Read from a named pipe or stdin instead of `fopen`, so it can tail a live log.
- **ncurses TUI.** Eventually. Cosmetic.

But that all comes after the OSTEP MapReduce project, which is the real next step. HiveParser was the warm-up. The patterns I worked through here — caller-holds-the-lock, condvar with done flag, per-thread accumulation merged at the end — are *exactly* the patterns the MapReduce assignment asks you to implement. That's why I built this first. Walking into MR_Run cold without having written a producer-consumer queue first sounded like a good way to get nothing done in two weeks.

## What I'd Tell Someone Building This

A few things, in no particular order.

**Use `while` for condvars, always.** Not `if`. Don't argue with yourself about it. OSTEP will tell you and you'll think you're smart enough to skip it. You're not. Nobody is.

**Run it under Helgrind early.** `valgrind --tool=helgrind ./hiveparser` will catch lock-ordering violations and missing locks before they catch you in production. The output is noisy and weird, but a real race condition stands out.

**Per-thread-then-merge beats one-global-with-a-mutex every time, until it doesn't.** For fixed-size accumulators like counters, there's no contest. The day you need a shared hashmap with millions of keys and you can't fit one per thread in memory is the day you start thinking about lock striping. That day is not today.

**Write the README first.** I didn't. Now I'm writing it after the fact and trying to remember why I made decisions I made three days ago. Future me hates past me, news at eleven.
